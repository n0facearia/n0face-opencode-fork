## Commit 1: Mode-Name Normalization Fix (v1.14.51)

### Bug
When a PCP (Pipeline CheckPoint) suggested `"frontend mode"` as next mode, the outer normalization at `pipeline.ts:290` converted it to `"frontend-mode"`, which failed `VALID_MODES.includes()` because `VALID_MODES` has `"frontend"` (no `-mode` suffix). This caused the Continue/Feedback dialog to silently skip, making it appear as if nothing happened after a mode completed.

### Root Cause
- PCP parser (`parsePipelineCheckpoint`) did NOT strip `\s+mode$` suffix — it only lowercased and trimmed.
- The outer normalization in `checkAndHandlePipelineCheckpoint` then replaced spaces with hyphens, producing `"frontend-mode"`.
- The catch-all at line 292 (originally `modeName.replace(/\-?mode$/i, "")`) did exist but ran AFTER the VALID_MODES check.

### Fix (3-part)
1. **PCP parser normalization** (line ~120): Added `.replace(/\s+mode$/i, "")` after the match, so `"frontend mode"` → `"frontend"` at parse time.
2. **Outer normalization catch-all** (line ~292): Added `.replace(/-?mode$/i, "")` as a safety net for any mode name that still has a `-mode` suffix.
3. **Exported `VALID_MODES`** from `pipeline.ts` and consolidated `scaffold.ts` to import from `pipeline.ts` instead of redefining its own list.
